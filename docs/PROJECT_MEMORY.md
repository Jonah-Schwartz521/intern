## Decisions Made

- **Framework: Tauri 2.x.** Chosen over Electron. Splerm is an always-resident hotkey/tray utility, the textbook Tauri-shaped product: lean footprint matters because it runs all day, and the Rust backend suits the audio/transcription work. Accepted tradeoff: to keep Rust exposure near zero, lean on official @tauri-apps plugins and the JS frontend, and treat "I need custom Rust" as a stop-and-ask signal. Windows-first sidesteps Tauri's main downside (per-OS webview rendering differences).
- **Name: Splerm** (renamed from Intern). The app was originally called Intern, chosen because no single product owned the name (InternLM is a model family, "AI intern" is common marketing copy but not a product) and it fit the junior-helper metaphor. Renamed to Splerm across the product name, window title, tray, system prompt, notifications, and UI copy. NOT changed, deliberately: the on-disk repo folder (`C:\dev\intern`), the bundle identifier (`com.jonah521.tauri-app`, so saved conversations keep their store path), the internal message-role enum value `"intern"` (serialized into every saved session; renaming it would break rendering of old conversations), and the internal-only identifiers `intern-voice-input` (temp file, scoped in a capability) and `InternReminder_` (schtasks task prefix).
- **Scope boundaries.** Slack/Teams sending, mutual-availability scheduling, autonomous execution without confirmation, and live real-time streaming transcription are OUT of MVP. Email moved IN scope this session as draft-and-handoff (Splerm creates a Graph draft, the user sends manually); autonomous email SENDING stays out. See the "Email drafting" decision below and NORTH_STAR.md.
- **Transcription order.** Start with file-based and playback-triggered transcription. Live streaming transcription is a later, bigger lift.
- **Architecture.** Single Claude call that routes intent to start. No internal sub-agents until the loop works. Orchestration lives in the frontend; reach into Rust only if a plugin genuinely cannot do the job.
- **LLM model routing + caching.** Use a smart model cost-effectively rather than picking a cheap-everything model. Default routine intent parsing to Haiku 4.5; escalate to Opus 4.8 only for ambiguous, multi-step, or context-heavy requests. Cache the static system prompt (instructions + tool definitions) so repeated input bills at the cache-read rate. Keep the model behind a thin abstraction (config value, not hardcoded). No Batch API (it is async, Splerm is real-time). Estimated cost at ~50 commands/day: roughly $2/month routed, ~$16/month if everything went through Opus. Rationale: intent parsing is mostly classification/extraction, which the cheap tier handles well; Opus is reserved for the requests that justify it.
- **Transcription: local whisper.cpp.** Run transcription on-device via whisper.cpp instead of a paid per-minute API. Zero per-minute cost, works offline, private. Pipeline: ffmpeg extracts audio -> whisper.cpp transcribes -> text. Shelled out from Tauri like ffmpeg. Optional cloud fallback (e.g. Groq Whisper) can be added later, not MVP. This was the cost line item that actually mattered (audio is billed per minute), so removing it removes the only real recurring cost.

- **Hotkey: Ctrl+Shift+Space.** Changed from the originally-planned Ctrl+Shift+I because Ctrl+Shift+I collides with the webview DevTools shortcut in dev mode (pressing it opened DevTools instead of firing the global shortcut). Space is clean, one-handed, no collision. Registered via CmdOrCtrl+Shift+Space in code (CmdOrCtrl resolves to Ctrl on Windows). NOTE: update NORTH_STAR.md, which still lists Ctrl+Shift+I.
- **Outlook calendar + mail integration (Microsoft Graph, PKCE).** Provider is Outlook via Microsoft Graph. Auth is the PKCE authorization-code flow for a public native client (no client secret). Current app: client ID `f4c142b4-85f2-4d7d-801d-5e8c59fe0676`, registered **Personal Microsoft accounts only**, so the auth endpoint is `https://login.microsoftonline.com/consumers/oauth2/v2.0` (NOT `/common/`, which requires an "All" audience). Redirect `http://localhost` (Mobile/desktop platform, loopback, port-agnostic), public client flows enabled. Scopes: `Calendars.ReadWrite Mail.ReadWrite User.Read offline_access` (Mail.ReadWrite for creating drafts; NO Mail.Send, sending stays manual in Outlook). See the app-registration saga under Blockers/Gotchas for why the client ID / endpoint changed.
  - **Three plugins added** (touches Rust config, flagged and approved): `tauri-plugin-oauth` (community, FabianLars) spawns a localhost server to catch the redirect; `@tauri-apps/plugin-http` for CORS-free token + Graph calls; `@tauri-apps/plugin-store` for token storage. All registered in lib.rs, deps in Cargo.toml, capabilities scoped in default.json.
  - **Why plugin-http and not browser fetch for token/Graph:** the Azure app is a NATIVE client. Microsoft only returns CORS headers on the `/token` endpoint for the SPA platform type, so a webview fetch would be blocked. plugin-http runs the request through Rust and sidesteps CORS. (The Anthropic call stays on browser fetch; it works via the dangerous-direct-browser-access header.)
  - **Loopback port:** the oauth plugin binds a random localhost port; Azure ignores the port on `http://localhost` loopback redirects, so it matches the registered redirect. The `redirect_uri` sent to authorize and token must be identical (`http://localhost:{port}`).
  - **Capabilities:** used `oauth:allow-start` + `oauth:allow-cancel` (auto-generated per-command perms, guaranteed to exist) rather than `oauth:default`, since the community plugin may not define a default set. `http:default` is URL-scoped to `login.microsoftonline.com/*` and `graph.microsoft.com/*` only.
  - **Could not compile-check Rust in the agent environment:** crates.io was unreachable (SSL reset) so the three new crates could not be fetched. Frontend `tsc` passes. First `npm run tauri dev` on a normal network downloads the crates and validates capabilities.
- **Transcription: Const-me/Whisper (GPU), no ffmpeg, everything bundled.** File transcription shells out to a bundled binary (no whisper-rs crate, keeping Rust near zero).
  - **Whisper engine: Const-me/Whisper**, a Windows-only fork of whisper.cpp that runs on the GPU via DirectCompute/Direct3D 11 (works on most GPUs, not just NVIDIA). Chosen for GPU speed on a Windows-first app. Pinned to release **1.12.0 (2023-07-22, its last release, stale but functional)**. The `cli.zip` asset contains `main.exe` (0.2 MB) + `Whisper.dll` (0.55 MB); the DLL must sit next to the exe. CLI is whisper.cpp-style: `-m <model>`, `-f <input>`, `-otxt` (writes `<input>.txt`, there is NO `-of`), `-nt`, `-l`, plus `-la` (list GPU adapters) and `-gpu`. Model format is standard ggml, so `ggml-base.bin` works.
  - **Accuracy note for future sessions:** the stated rationale was "upstream removed direct binary downloads," but Stage 0 verification found ggml-org/whisper.cpp DOES still ship Windows prebuilts (v1.9.1, `whisper-bin-x64.zip`, with a `whisper-cli.exe`). Const-me was chosen anyway for GPU acceleration; recorded so we don't re-derive this.
  - **ffmpeg: DROPPED (Stage 3 verified not needed).** Const-me `main.exe` decodes raw `.wav`, `.mp3`, AND `.mp4` (video, audio track pulled) directly via Windows Media Foundation. Test: all three transcribed with exit 0, each logging "Created source reader from the file ...". So there is NO ffmpeg conversion step and NO ffmpeg bundle (~80 MB and the gyan `.7z`-extraction problem both avoided). Revisit only if some future input format fails MF decode.
  - **Transcript capture: STDOUT.** `main.exe -m <model> -f <input> -nt` prints the clean transcript to stdout; all GPU/perf logging goes to stderr. Capture `Command.execute().stdout` — no `-otxt`, no temp output file. Verified: jfk.wav → "And so my fellow Americans ask not what your country can do for you, ask what you can do for your country."
  - **Perf (Intel Arc 140T GPU):** an 11 s clip ran in ~1.1 s total (0.5 s model load + 0.6 s GPU inference). The "30-60 s for 10 min" caveat was a CPU-era worst case; GPU is far faster.
  - **Model: `ggml-base.bin`** (~142 MB, multilingual) is BUNDLED as a Tauri resource, NOT downloaded. First-run download was built and then abandoned: Hugging Face TLS-reset intermittently on the target network across BOTH `curl` AND PowerShell `Invoke-WebRequest`. The reset is network-level, so the curl->IWR swap was a dead end. Bundling is deterministic and offline (installer bloat accepted). Copied from the resource into app-data on first use with an exact-byte-size integrity guard (147,951,465 bytes) so a truncated copy can never poison later runs.
  - **Binaries bundled as resources, resolved via the `$RESOURCE` scope variable (NOT JS `resolveResource`).** `main.exe` + `Whisper.dll` (~0.75 MB, co-located in `src-tauri/resources/whisper/`, committed) plus the model ship via `bundle.resources`. The `whisper-run` shell scope `cmd` is `$RESOURCE/resources/whisper/main.exe` - Tauri v2 shell scope supports the `$RESOURCE` variable, and `Command.create(name)` matches the scope by name and executes its `cmd`, so path resolution belongs in the scope, not JS. Whisper.dll co-locates with the exe in the same resource folder. Works in dev and installed builds.
  - **plugin-dialog** (file picker for Transcribe file) was installed, reverted, then re-added for the wiring stage. Now in place.
- **Voice input records `audio/mp4`, not MediaRecorder's default `webm/opus`.** Mic capture uses `getUserMedia` + `MediaRecorder` with an explicit `audio/mp4` mimeType. Reason: whisper's decoder is Windows Media Foundation, which decodes AAC (in mp4) but NOT WebM/Opus. Recording mp4/AAC means MF reads the file directly with no converter step (falls back to the browser default only if a machine cannot record mp4, and warns). Verified in Stage 2: `MediaRecorder.isTypeSupported("audio/mp4")` is true on this machine and the mp4 clip transcribed. Voice transcripts go to the input box (a command to review and send); an empty/too-short capture (< 2 KB) is skipped before hitting whisper.
- **System audio capture via Stereo Mix.** "Transcribe what's playing" taps the Windows Stereo Mix loopback device: enumerate audio inputs, match `/stereo mix/i`, pass its browser `deviceId` to `getUserMedia({ audio: { deviceId } })`, then the same mp4 -> whisper path. This is the clean digital loopback (works with the system muted or on headphones), unlike an acoustic mic pickup. System-audio transcripts render as a transcript bubble (content), while mic transcripts go to the input box (command). CAVEAT: Stereo Mix ships DISABLED by default on most Windows machines and must be enabled manually in Sound settings - fine for personal use, a blocker for distribution without a manual-enable step.
- **Email drafting (moved IN scope; was NOT-MVP).** Splerm drafts and CREATES an Outlook draft via Graph (`POST /me/messages`, Mail.ReadWrite), then opens it via its `webLink` in browser Outlook. Deliberately NO `Mail.Send` - draft-and-handoff, the user reviews and sends manually. UX is an inline prefilled compose card ("Model B"): the `draft_email` tool has Claude generate To + a specific Subject + a full first-pass Body up front, the card renders in the chat stream (not a modal) with all three editable, and Create draft posts the edited field values. Edit-don't-interrogate: always draft, never ask field by field; only ask first if the recipient is genuinely unresolvable (then still draft subject+body and leave To blank). Single natural-language path - an earlier blank "Draft email" overflow-menu button was added then removed because a requestless button can only open an empty card. Email-drafting requests escalate to Opus in `pickModel` because Haiku under-fills bodies. (Graph draft code in `src/mail.ts`.)
- **Dev environment: Windows-native only.** Project lives at C:\dev\intern. A duplicate copy existed under WSL (\\wsl.localhost\Ubuntu\...) and caused repeated failures: Windows npm choking on Linux paths (EISDIR/EPERM), CMD rejecting UNC paths, stale edits landing in the wrong copy. WSL copy deleted. Rule: only ever work from PS C:\dev\intern>, never a wsl.localhost path. Partially resolves the "target OS first" open question (committed to Windows-first dev).

- **Conversation persistence + /resume (store plugin, no Rust).** Conversations are serialized to `sessions.json` via `tauri-plugin-store`, keyed by id, with a `currentId` pointer (`src/session.ts`). The current session restores on launch and saves on every change; empty conversations are never written and no-op renders are skipped via a last-saved snapshot so merely relaunching does not bump a session's timestamp. Slash input (`/resume`, `/new`) routes through a small command registry in `App.tsx` instead of Claude, so new commands are one map entry. The `/resume` list renders inline in the stream (no modal, per the app's pattern). CRITICAL invariant: every conversation switch (new, resume, load-a-row) calls `persistCurrent()` BEFORE clearing the view, so in-progress work is never lost. Titles: generated by ONE cheap Haiku call (`generateTitle`, no tools, no cached system prompt, `max_tokens: 16`, only trimmed opening turns sent) once a conversation hits 3+ messages, cached on the session and never regenerated; under 3 messages stays on the first-message placeholder so throwaway chats do not clutter the list or cost anything. Race guarded: the async title-save writes the LATEST messages (via `messagesRef`), not the stale snapshot that triggered generation, so a reply landing mid-request is not clobbered.

- **`/clear` deletes, it does not just forget.** `/clear` is the throwaway-conversation escape hatch: wipe the view and keep it out of the resume list, as opposed to `/new`, which persists first. Non-obvious part: autosave has usually ALREADY written the conversation to `sessions.json` by the time the user clears it, so dropping the in-memory id is not enough. `clearConversation()` calls `deleteSession(id)` to remove the record, then mints a fresh session. Because it is unrecoverable, it asks first via an inline confirm bar in the stream (no modal, per the app's pattern); Cancel, sending a message, and every conversation switch clear the pending confirm so it cannot go stale and fire against the wrong conversation. The unknown-command notice now lists the registry keys instead of a hardcoded "/resume or /new", so it stays correct as commands are added.
- **Clipboard as a first-class input (`read_clipboard`).** The clipboard is the zero-friction input for a hotkey tool: copy an error / JSON / paragraph, summon Splerm, say "explain this". Implemented as a normal tool in the loop (`@tauri-apps/plugin-clipboard-manager`, official, so no custom Rust; read-only capability `clipboard-manager:allow-read-text`, no write). The model-side half is a system-prompt rule: a demonstrative ("this", "that") with no referent in the conversation means the clipboard, so call the tool and never ask the user to paste; if the referent IS in the conversation (a file just found, an event just listed) the tool is skipped. Empty or non-text clipboards return a one-line instruction to the model, not an error dump (the plugin throws on images/files, which is expected, not a failure). Clipboard text is truncated to 8,000 chars so a stray copy of a huge document cannot blow up the bill. Output side: `takeawayContent()` gives a reply that is essentially ONE fenced code block (>= 60% of the reply) the existing copy-bubble treatment, so reformatted JSON or rewritten text is one click from going back to the clipboard, while prose answers stay unboxed. `max_tokens` was raised 1024 -> 2048 because reformatting echoes the payload back and 1024 truncated mid-size JSON (output bills per token generated, so this is free for short answers).
  - **Two failure modes found in testing, both prompt-side, both fixed by wording.** (1) The tool fired inconsistently: the same "format this" asked "what do you want me to format?" on one attempt and worked on the next. A politely-phrased rule is not enough for the small model, so the rule is now an explicit hard rule (demonstrative + no antecedent = ALWAYS call read_clipboard, never ask what they mean, never ask them to paste), with the trigger phrases listed. (2) A reply once claimed "I don't see new content on your clipboard". There is NO caching or change detection in the code (`readText()` is a fresh plugin call, `runTool` holds no state); the model invented it, most likely because an earlier clipboard read sits in the conversation as a tool result and it started reasoning about old-versus-new. Fixed by stating statelessness in three places: the tool description ("no cache, no history, no change detection, it cannot tell you whether the contents are new"), a system-prompt rule (never say the clipboard is unchanged or has nothing new; act on the text even if identical to an earlier read), and the tool result string itself ("Clipboard contents, read just now. Act on this:").
- **Spotlight-style summoning (JS window API, no Rust).** The hotkey positions the window before showing it (`src/summon.ts`): centered horizontally, top edge 22% down the monitor's `workArea` (upper third reads as a launcher; dead center looks low and heavy), on the display under the mouse cursor via `cursorPosition()` + `monitorFromPoint()`, falling back to `currentMonitor()` then `primaryMonitor()`. Monitor geometry and `outerSize()` are both PHYSICAL pixels, so they compare directly with no scale-factor math; `workArea` (not `size`) is used so the window clears the taskbar on any DPI. `setPosition` runs before `show()` so the window never appears at its old spot and jumps, and the whole positioning block is wrapped in try/catch so a failure still summons the window. Focus: the webview only takes keyboard focus after the OS hands the window focus back, so `focusInput()` calls `focus()` immediately and again on the next frame. All the reads (`cursor-position`, `monitor-from-point`, `current-monitor`, `outer-size`) are already granted by `core:window:default`; the only new capability is `core:window:allow-set-position` (in `capabilities/desktop.json`). KNOWN GAP: the tray's "Show Splerm" item shows the window from Rust (`lib.rs`) and does NOT reposition; routing it through the same path would need Rust to emit an event, so it was left alone.

- **Context bin (data + ingestion layer, no UI).** User-uploaded reference files fed to the model as steering context. `src/contextBin.ts` owns records + persistence + the public API (`addFiles`, `removeFile`, `listFiles`, `getTotalTokens`, `pickContextFiles`); `src/extract.ts` owns text extraction. Record shape: `{ id, filename, path, mimeType, extractedText, charCount, tokenCount, addedAt }`.
  - **Storage: tauri-plugin-store, NOT tauri-sql.** The bin is a small set (a handful of reference files), loaded all at once, never queried relationally, and the whole app already standardizes on the store plugin (`sessions.json`). SQL would add a Rust dep + migrations for no benefit at this scale. Lives in `context-bin.json`, key `files` (id -> record), mirroring `session.ts` (lazy singleton handle, `autoSave: false`, explicit `save()`).
  - **Extraction, all in the webview (no Rust).** `.txt`/`.md`: `readFile` bytes + `TextDecoder` (using readFile not readTextFile so the module needs only `fs:read-file`). `.pdf`: `pdfjs-dist`, page-by-page text layer. `.docx`: `mammoth` (`extractRawText`). Unsupported types skip with a reason; a supported file that yields empty text (blank file, or a scanned/image-only PDF with no text layer) also skips with a reason. OCR for image-only PDFs is a deliberate non-goal (TODO if needed).
  - **Two libs added:** `pdfjs-dist`, `mammoth`. pdf worker wired via Vite `?url` import (`pdfjs.GlobalWorkerOptions.workerSrc`). mammoth imported from the package ROOT (`import mammoth from "mammoth"`) not the `/mammoth.browser` subpath: root has the types (`lib/index.d.ts`) and Vite auto-applies mammoth's `browser` field (swaps unzip/file internals for browser-safe ones). Verified: tsc clean + a lib-mode Vite bundle of contextBin pulls both in and builds (374 modules, ~2.97 MB).
  - **Capability, flagged:** reading a user-picked file needs `fs:allow-read-file`. A reference file can live anywhere the user browses (Documents, but also project dirs like `C:\dev`), so read scope is `**` by necessity, read only (no write). Narrow to `$HOME/**` if files will only ever come from home. Added to `capabilities/desktop.json`. Tauri v2's dialog does NOT auto-grant fs scope to picked paths, hence the explicit read capability.
  - **Token count is `Math.ceil(chars/4)`**, a cheap first pass. TODO left in code to swap in a real tokenizer (e.g. `@anthropic-ai/tokenizer`) so counts match billed tokens.
  - **Re-adding a path replaces its record in place** (same `id`, original `addedAt` preserved) so an edited file is refreshed by adding it again, and downstream references to the id stay stable.
  - **Wired into the Claude request path (`askClaude` in App.tsx).** `assembleContextBlock()` wraps each file as `<context_file name="{filename}">\n{extractedText}\n</context_file>`, joined by blank lines. Injected as a SECOND `system` text segment after `SYSTEM_PROMPT`, each with `cache_control: ephemeral`. KEY caching fact: the Anthropic API builds its cache prefix in a FIXED order (tools -> system -> messages) no matter how our code is ordered, so tool defs physically precede the base-instructions text even though the request logically reads "base instructions -> tool defs -> context block". What matters holds: context block is the LAST cached segment and the per-call user message (in `messages`) is never cached. Two breakpoints on purpose: breakpoint A (on base instructions) keeps tools+base cached when the bin changes; breakpoint B (on context block) extends the cache, so adding/removing a file only invalidates from B onward. Empty bin => no system[1], identical shape to pre-feature (one breakpoint). Assembled ONCE per turn, outside the tool-use while loop.
  - **Budget guard, not a hard block.** `assembleContextForRequest(budget = DEFAULT_CONTEXT_TOKEN_BUDGET)` (default 50000) returns `{ block, totalTokens, budget, overage }`. Over budget: `console.warn` + a warning through the new module-level `uiWarn` sink (routed to the same inline `notice` bar as command notices), then send anyway. Deliberately never silently ships a bloated bin, but never blocks the user either.
  - **Model routing untouched.** The bin is part of the shared `system`, so it applies to whichever model `pickModel` picks (Haiku or Opus).
  - **UI: collapsible Context panel (`ContextPanel` in App.tsx), frontend only.** Sits as a `flex: 0 0 auto` section BETWEEN the titlebar and `.history` (which is `flex: 1`, so it absorbs the height; no existing layout properties touched). Not a modal, not a separate view, matching the one-window pattern. Head is always visible (label "Context" in sans + running total token count in mono); body toggles open/closed. Token total color: neutral (dim) with headroom, amber (copper) at >= 80% of `DEFAULT_CONTEXT_TOKEN_BUDGET`, red (#d9776b) once over (visual only, the prompt-path guard still sends). File rows: filename mono + ellipsis (title=full path), token count mono, remove (x). Empty state: one sans line.
  - **Drag-and-drop uses Tauri's native `onDragDropEvent`, NOT DOM drop.** `dragDropEnabled` defaults to true in Tauri v2, so the webview suppresses HTML5 drop events and the OS handles file drops, emitting `tauri://drag-drop` with REAL filesystem paths (DOM drop in a webview does not expose paths). A drag over the window auto-opens the panel and highlights the dropzone (`dragOver` state); the drop feeds `p.paths` straight into the shared `ingestPaths` -> `addFiles`. Listener registered once on mount via `getCurrentWindow().onDragDropEvent`; covered by `core:default` (event listen), no new capability.
  - **Design language:** added a `--sans` CSS var (`"Segoe UI", system-ui, ...`, Windows-first) because the app body font is mono globally; the panel uses sans for UI labels/buttons/empty-state and mono for filenames/paths/token counts, per NORTH_STAR's "mono for code/paths, sans for UI text". Two deliberate uses of reserved layout props on NEW isolated elements, flagged per CLAUDE.md: `overflow-y: auto` + `max-height` on `.ctx-list` (so a big bin scrolls instead of pushing the input off-screen) and `overflow: hidden` + `text-overflow: ellipsis` on `.ctx-file-name` (same pattern already used by `.oa-email`). No changes to `position`/`z-index`/`background-color` on existing elements.
  - **NOT verified in the running GUI.** tsc + `vite build` pass; the Tauri window was not launched/screenshotted in this environment. First `npm run tauri dev` should confirm the panel visually and exercise a real drag-drop.

- **Chat code blocks: Shiki syntax highlighting + language label.** `src/highlight.ts` + a `CodeBlock` component in App.tsx (rendered via a react-markdown `pre` override).
  - **Shiki, not highlight.js:** real VS Code TextMate grammars + themes, so code matches the Monaco/VS Code look. Theme `github-dark` (carries its own background, reads well on the warm chat; chosen over dark-plus).
  - **One highlighter, lazy grammars.** A single highlighter is created lazily (`createHighlighter({ themes:['github-dark'], langs:[] })`) and reused; each grammar loads on first use via `loadLanguage` and is remembered in a `loaded` set. Vite code-splits every Shiki grammar and the oniguruma wasm engine into their own chunks (confirmed in the build output: `typescript-*.js`, `cpp-*.js`, `wasm-*.js`, ...), so the initial bundle does not carry 235 languages and startup stays snappy. The wasm loads only when the first code block renders; `tauri.conf.json` has `csp: null`, so `WebAssembly.instantiate` is allowed.
  - **Language detection + fallback.** Language comes from the markdown fence class (`language-xxx`). `isSupportedLang` checks a Set built from `bundledLanguagesInfo` (ids + aliases, so `ts`/`py`/`sh` work). If the fence has no language, or one Shiki has no grammar for, the block renders plain (no highlight) and the label is hidden. Never guesses a language. Highlight is async; the plain fallback shows until it resolves, and an `alive` flag drops a stale result.
  - **Header row per block.** Language name top-left (mono, copper `#c07840`, small/subtle), Copy top-right (reuses `.copy-btn`; `.code-copy { margin-left:auto }` keeps it right when no label). `overflow-x: auto` on `.code-block pre` (long lines scroll, VS Code style) is a deliberate reserved-prop use on a new element, flagged; Shiki's own inline `background-color` is left untouched (no CSS background-color edits).
  - **Removed `takeawayContent` and the message-level copy for code replies.** Previously a reply that was ~one fenced block got a `.card` box + a top-of-message Copy (`takeawayContent`). With per-block headers that would double up (nested box, two Copy buttons), so the reply push no longer sets `copyText` and `takeawayContent` was deleted. Transcript copy bubbles (voice/system/file) set `copyText` explicitly and are unchanged.
  - **`pre` override, not `code`.** Block code is always the single `<code>` child of a `<pre>`, so overriding `pre` cleanly leaves inline `` `code` `` on the default renderer. Code text is read from the child's `children` (array-joined defensively) with one trailing newline stripped.
  - **NOT verified in the running GUI.** tsc + `vite build` pass and the Shiki logic (alias resolution, unknown-lang fallback, real highlighting, lazy-load of only used grammars) was verified in Node. The webview rendering (label layout, injected Shiki HTML, wasm load in WebView2) was not launched here; confirm on first `npm run tauri dev`.

- **Edge-docked slide-in panel (replaced floating window + `summon.ts`).** Window shell changed from a spotlight-style floating window to a sidebar docked flush to a screen edge. Pure Tauri, no custom Rust (a hard project constraint).
  - **Why transparent + the reserved-prop touches were unavoidable, and were approved.** Rounding only the INNER corners (so the panel reads as docked, edge-side square) means the corner gaps must reveal the desktop, which requires `transparent: true` on the window + `background: transparent` on `body` (surface color moved to `.container`) + `overflow: hidden` on `.container` to clip the rounding. Those touch the `background-color`/`overflow` props CLAUDE.md says not to touch without an explicit ask; the user was asked (transparent-window option) and chose it. No `position`/`z-index` changes; the slide is `transform` only, so layout is never touched.
  - **Physical-px math, DPI-safe.** `dock.ts` works entirely in physical px (what `monitor.workArea` reports), so `setSize`/`setPosition` need no scale-factor juggling; the ONE logical value, `MIN_SIZE_PX` (360 CSS px), is converted to physical via `monitor.scaleFactor`. ## Blockers / Gotchas

- **Azure app-registration saga (AADSTS70000 incremental consent → audience mismatch).** Adding Mail drafting required Mail.ReadWrite, which set off a chain of auth problems worth recording so we never repeat it.
  - **Stranded original app.** The original registration (client ID `c44b4083` per Jonah's Azure notes; the ID that had been in the codebase was `233ef4ee`, swapped out in commit history) lived in the **Comcast default-directory tenant**, which no accessible account could edit. So Mail.ReadWrite could not be added to it. Consenting to a scope the app doesn't expose caused **AADSTS70000** on the next token refresh (the refresh_token was for a scope set the app couldn't grant).
  - **Incremental-consent trap.** With `prompt=select_account`, reconnecting reused cached consent and silently skipped granting the newly-added scope. `prompt=consent` forces the screen and re-grants, but that was a workaround, not the fix.
  - **Real fix: fresh registration under the personal account.** Registered a new app (`f4c142b4-85f2-4d7d-801d-5e8c59fe0676`) under the personal outlook.com account (`jonah.schwartz5`) via entra.microsoft.com, with all four Graph scopes at registration (`Calendars.ReadWrite`, `Mail.ReadWrite`, `User.Read`, `offline_access`), redirect `http://localhost` under Mobile/desktop, and "Allow public client flows" = Yes (pure PKCE, no secret).
  - **Audience mismatch: `/consumers/` not `/common/`.** The new app is **Personal-accounts-only** (Consumer audience). `/common/` requires an "All accounts" audience and fails with an audience mismatch. Switched `AUTHORITY` in msauth.ts to `https://login.microsoftonline.com/consumers/oauth2/v2.0`. Reverted `prompt` back to `select_account` since the fresh app has both scopes from the start (no forced consent needed).
  - **Lessons:** register the Azure app under the personal MS account that actually owns the calendar/mail (not an org/Comcast default tenant); add ALL needed Graph scopes at registration up front (avoids AADSTS70000 incremental-consent pain); match the auth endpoint to the app's audience (`/consumers/` for personal-only, `/common/` only for multi-audience apps).
- **AADSTS90023 on the Microsoft token exchange (Origin header).** The `/token` call failed with AADSTS90023 (cross-origin token redemption) even though it goes through `@tauri-apps/plugin-http` (Rust-side, not browser fetch). Cause: plugin-http's Rust `commands.rs` auto-injects an `Origin` header derived from the webview URL (`http://localhost:1420` in dev, `tauri://localhost` in prod). Microsoft rejects any token request carrying Origin unless the app is registered as an SPA. Fix, two parts, BOTH required: (1) build the crate with the feature: `tauri-plugin-http = { version = "2", features = ["unsafe-headers"] }`; (2) send `Origin: ""` in the request headers. The plugin only strips Origin when unsafe-headers is on AND Origin is explicitly empty. `Origin: ""` survives the JS `new Headers(obj)` path because a directly-constructed Headers has guard "none" (forbidden-header filtering only applies to Headers taken from a Request). Both token calls use the shared `TOKEN_HEADERS` const in msauth.ts. Apply the same `Origin: ""` to future Graph calls if any endpoint objects to it.
